# Design Decisions Record (DDR) - 萱草日签：帮助与共创中心

本文档记录了 `07-help-feedback.html`（帮助与反馈）和 `08-feature-voting.html`（功能许愿池）的设计共识。这两个页面承担了产品闭环中的“听取用户声音”和“陪同产品共建”职能。

## 1. 帮助与反馈：极简克制与降噪设计
**[决策背景]**
长辈或是时间紧迫的照护者在遇到使用障碍时，往往缺乏耐心。复杂的客服系统、层层嵌套的知识库导航会让他们迅速放弃寻求帮助。本产品必须提供阻力趋近于零的求助通路。

**[设计共识]**
- **高曝光、低密度**：摒弃长篇大论的帮助文档集，采用单层列表形式，刻意限制并只展示当下时段最高频的 3 个问题（如邀请家人、忘记打卡、PRO影子账户等），其余一律留白。
- **宽容的倾诉空间**：反馈区摒弃了复杂的“工单分类（Bug/建议/充值）”和“联系方式必填”等表单验证。直接以一个带超大圆角的留白输入框作为“唯一倾听口”，减轻一切输入前的门槛。
- **同理心辅助**：“上传截图”被具象化为一个带有弱虚线边框的拟物小方块，搭配“提供截图帮助我们快速定位”这句文案，极大地降低了用户对于“提交日志”方面的技术恐慌。

## 2. 功能许愿池：赋予参与感与情感化决断
**[决策背景]**
对于 C 端普通家庭用户而言，“产品需求池”、“用户共创平台”这类互联网黑话显得过于生硬冰冷，并不能唤醒他们参与产品建设的热忱。

**[设计共识]**
- **充满陪伴感的文案包装**：彻底弃用“需求”，将其更名为“功能许愿池”，并在页面顶部引入“🌱 一起种下未来”的标语，意在向用户传达“你不仅在使用，更在亲手建设这个家园”的温馨价值。
- **极其收敛的决策漏斗（2选1）**：无论产品侧收集了多少需求，前端永远只向用户抛出当前版本最需要纠结的 **2 个**待选功能（例如：扫码智能添药 vs 家庭周报）。以占据半屏的大尺寸面性卡片去承载，免除长列表对比带来的选择困难症。
- **高反馈值的游戏化结算**：摒弃了点个赞就拉倒的敷衍设计。用户点击即代表一锤定音：点击的瞬间，背后的选项直接隐身，界面过渡到全屏的“战况结算面板”。柱状图在 1 秒内从 0 飙升到最终百分比的动效，旨在向用户传递最浓烈的视觉多巴胺——“你的这一票，真的主宰了产品的走向”。
