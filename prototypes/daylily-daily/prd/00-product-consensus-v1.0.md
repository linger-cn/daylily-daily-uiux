## 2. One-Line Summary

`一款拒绝臃肿、不追求医疗级（HIS）完备性、主打“极简依从性追踪”的轻量级微信小程序。核心利用家庭社交链条解决长辈用药提醒与监督的痛点。`

## 3. Core Positioning

### 3.1 Product Positioning

本产品的核心不是做一个“大而全的医疗管理系统”，而是在微信生态内做一个足够轻、足够稳、足够低门槛的依从性追踪工具。

在功能层面上坚决做减法：
1. 以打卡日历作为唯一核心主态
2. 优先解决“提醒是否到达、是否执行、异常如何升级”
3. 将“温情周报”作为后续社交裂变钩子，而不是一开始堆复杂功能

### 3.2 What We Intentionally Are Not

本产品明确不做以下方向：
1. 不追求医疗级（HIS）完备性
2. 不对接庞大的专业药品库
3. 不做复杂 OCR 识别
4. 不做极易失准并反噬体验的“药盒库存倒计时”
5. 不做高违规风险且体验过重的“微信个人号群聊机器人”
6. 不做复杂的影子账户数据合并
7. 不做面向长辈的无意义 UI 颜色皮肤或装扮型功能

## 4. Background and Why Now

### 4.1 Industry / User Context

这个产品成立的前提不是“医疗信息管理不够强”，而是“家庭里的提醒与监督闭环不够顺”。

长辈用药往往发生在高频、琐碎、家庭化的日常场景里。真正承担监督成本和焦虑的，往往是家庭中的照护者，而不是专业医疗系统本身。

### 4.2 Why This Problem Matters Now

微信仍然是最现实的家庭触达与协作基础设施。

如果能够在微信生态内，用足够轻的方式完成“提醒 -> 打卡 -> 漏服升级 -> 家庭触达”闭环，就有机会避开重医疗系统的复杂性，同时解决一个真实、长期、反复发生的问题。

## 5. Target Users

### 5.1 Primary Users

本产品天然是双边核心用户结构：
1. 被照护者：需要被提醒服药、完成打卡的长辈或患者
2. 照护者：负责创建计划、监督执行、补救漏服的子女、配偶或核心家属

### 5.2 Secondary Users

次级用户包括：
1. 没有照护者、但有自我管理需求的个人用户
2. 无智能手机或不会使用小程序的老人，由照护者代为管理与代打卡

### 5.3 Decision-Maker / Payer

如果存在明确的决策者与付费者，默认应是照护者，尤其是子女。

原因不是其操作更多，而是其更有购买力，也更在意工具是否能降低照护焦虑、提供情绪价值和家庭掌控感。

### 5.4 Care Group Model

关系边界采用“照护组（Care Group）”概念进行管理：
1. 支持多对多关系
2. 一个用户可同时身处多个组
3. 用户在不同组内的角色可以不同

示例：
1. 在父母组是照护者
2. 在夫妻组是被照护者

同时，产品完美兼容没有照护者的个人用户，即“创建者”与“执行者”为同一人的单机闭环。

## 6. Core Problem Statement

### 6.1 Current User Pain

当前用户痛点集中在以下几类：
1. 长辈服药是高频日常动作，但现有工具往往过重、过复杂
2. 被照护者不一定会稳定使用小程序或完成授权，提醒链路容易断
3. 家庭监督通常依赖人肉记忆或零散沟通，缺少清晰的执行回执与升级机制
4. 真正有监督意愿的家属，缺少一个足够轻量、足够低摩擦的协同工具

### 6.2 Existing Alternatives and Their Gaps

当前替代方案存在明显不足：
1. 医疗级产品太重，不适合家庭日常依从性追踪
2. 药典、OCR、库存等复杂能力会快速拉高使用门槛
3. 普通提醒工具只能提醒，不能形成家庭监督闭环
4. 高风险微信机器人方案虽然“看似强大”，但不适合做稳定、合规、产品化落地

### 6.3 Problem Statement

本产品当前要解决的问题可归纳为：
1. 如何让被照护者或照护者以最低摩擦完成每日用药执行确认
2. 当长辈未授权或漏服断链时，如何借助家庭社交链条完成最终触达
3. 如何在微信生态内建立一个边界清晰、可信、轻量的家庭监督闭环

## 7. Product Goals and Non-Goals

### 7.1 Core Goals

1. 用极简交互完成“计划创建 -> 提醒 -> 打卡 -> 追踪”闭环
2. 让照护者在关键异常时刻拥有明确、可执行的监督与补救动作
3. 利用微信生态与家庭社交链条，提升长辈用药依从性

### 7.2 Non-Goals

1. 不做专业医疗管理平台
2. 不追求药学信息完整性与数据库完备性
3. 不追求复杂协同编辑、复杂库存和复杂机器人能力

## 8. Core Scenarios

### 8.1 Scenario A：照护者为长辈创建用药计划

- Trigger: 家属发现长辈需要稳定服药，但执行依赖口头提醒
- User: 照护者
- Goal: 快速创建一个自己和长辈都能理解的用药计划
- Current Friction: 现有工具录入复杂、字段太多、药品信息门槛高
- Expected Product Support:
  - 极简文本录入
  - 无需药典与 OCR
  - 只要求“创建者自己能看懂即可”

### 8.2 Scenario B：被照护者完成每日打卡

- Trigger: 到达服药时间，用户需要快速确认“我吃过了”
- User: 被照护者
- Goal: 用最低摩擦完成打卡
- Current Friction: 复杂步骤会降低打卡完成率
- Expected Product Support:
  - 首页核心区域提供巨大的“我吃过了”按钮
  - 支持通过微信提醒卡片快捷唤起打卡
  - 用户每次打卡时顺势续杯第二天的订阅消息授权

### 8.3 Scenario C：长辈漏服或提醒断链，照护者接手补救

- Trigger: 长辈未授权、未响应或出现漏服断链
- User: 照护者
- Goal: 快速完成最终触达和监督
- Current Friction: 系统通知本身不足以形成家庭监督压力
- Expected Product Support:
  - 系统将“漏服警告”发给照护者
  - 照护者可在小程序内点击【一键提醒】
  - 自动生成专属微信卡片，转发至家庭微信群或父母私聊
  - 利用真实亲情社交压力完成最终触达

### 8.4 Scenario D：无智能手机或不会使用小程序的老人

- Trigger: 被照护者无法独立完成打卡
- User: 照护者
- Goal: 代为维护连续性，而不引入复杂迁移问题
- Current Friction: 老人后续介入时，历史代管数据容易引发合并复杂度
- Expected Product Support:
  - 允许照护者代打卡
  - 支持影子账户代管
  - 一旦真实用户后续介入，系统以真实用户的新建进度为准
  - 坚决不做复杂数据合并（No-Merge）

## 9. Core Value Proposition

### 9.1 User Value

对被照护者而言，价值在于：
1. 录入极简
2. 打卡极简
3. 不需要理解复杂的医疗管理系统

对照护者而言，价值在于：
1. 可以清楚知道是否执行
2. 异常时有明确补救动作
3. 不再完全依赖人肉记忆和碎片化沟通

### 9.2 Family / Social Value

本产品的核心差异不是“提醒更花哨”，而是把家庭社交链条纳入监督闭环。

这条链条包括：
1. 漏服告警发给照护者
2. 照护者一键转发到家庭微信群或私聊
3. 后续可在产品内形成“家庭小客厅”式的留言和点赞

因此，它不仅是一个提醒工具，也是一种家庭责任协作工具。

### 9.3 Product Advantage

相比替代方案，本产品的优势在于：
1. 比重医疗方案更轻
2. 比普通提醒工具更有监督闭环
3. 比高风险机器人方案更可产品化、可合规落地
4. 在权限、隐私和代管边界上更清晰

## 10. Product Principles

1. 极简依从性追踪优先于功能堆砌
2. 去药典化，手工录入优先
3. 一键打卡优先于复杂交互
4. 谁建谁管，权限边界严格
5. 被照护者可拥有完全私密的个人计划
6. 影子账户只用于代管，不做复杂合并
7. 微信生态内的产品化路由优先于高风险外挂方案
8. 功能层面坚持做减法

## 11. Scope Boundary

### 11.1 In Scope

1. 照护组与单机个人模式并存
2. 极简手工录入的用药计划
3. 一键打卡与打卡日历
4. 订阅消息授权续杯
5. 漏服告警到照护者
6. 照护者一键提醒并转发到微信群或私聊
7. 被照护者私密计划
8. 小程序内留言 / 点赞等轻量家庭互动

### 11.2 Out of Scope

1. 医疗级（HIS）完备能力
2. 专业药品数据库与复杂 OCR
3. 药盒库存倒计时
4. 高违规风险的微信个人号群聊机器人
5. 跨照护者的复杂协同编辑与权限冲突处理
6. 影子账户与真实账户的复杂数据合并

## 12. Monetization and Growth Hypotheses

### 12.1 Monetization Direction

商业化默认瞄准真正具有购买力且需要情绪价值的照护者，而不是长辈本人。

### 12.2 Candidate Paid Package

“亲情定制包”是当前最明确的微创新方向：
1. 自定义专属语音：允许照护者用家人声音替代系统播报
2. 家庭相册背景：允许上传全家福作为打卡页半透明背景

这些能力不是“传统皮肤”，而是把工具转化为更具情感温度的产品界面。

### 12.3 Growth / Conversion Direction

另一个方向是基于家庭结构与用药依从标签，在周报或特定场景下，为照护者推荐高客单价服务，如深度定制体检或特定补充商业医疗险，走 CPS 模式。

该方向目前仍属于增长与商业化假设，不应先于核心闭环验证落地。

## 14. Key Constraints

### 14.1 Product Constraints

1. 产品必须保持低认知负担，尤其面向长辈使用
2. 首页交互必须足够直接，不允许为“设计完整性”牺牲打卡摩擦
3. 家庭协作必须保持边界清晰，避免复杂协同冲突

### 14.2 Technical Constraints

底层架构遵循以下共识：
1. 绝不将微信 UnionID 或 OpenID 作为系统业务主键
2. 从 Day 1 起建立独立 ID Mapping 体系，生成系统全局唯一 Global_User_ID
3. 高频“打卡（Punch）”视为不可变事件流（Event Stream）异步落盘
4. 逻辑上预留“高频打卡写入”与“日历 / 周报查询统计”的读写分离
5. 影子账户仅用于代管，不为未来埋下复杂数据合并包袱

### 14.3 Compliance / Trust Constraints

1. 高违规风险的微信个人号群聊机器人方案明确放弃
2. 被照护者的私密计划应完全不对照护者可见
3. 群内互动应以照护者主动转发为前提，而不是系统直接打扰群聊
4. 与 AI 生成分享卡相关的能力，必须谨慎处理合规、审美和信任边界

## 15. Assumptions and Risks

### 15.1 Key Assumptions

1. 家庭社交链条能显著提升最终触达与监督效率
2. 用户能接受“创建者自己看得懂即可”的极简药品录入方式
3. 照护者比被照护者更具付费意愿，也更需要情绪价值

### 15.2 Major Risks

1. 订阅消息授权续杯不稳定，可能导致提醒链路断裂
2. “人肉路由”虽然合规，但可能增加照护者操作负担
3. AI 生成卡片若处理不当，可能沦为噱头，甚至引发审美疲劳或信任问题
