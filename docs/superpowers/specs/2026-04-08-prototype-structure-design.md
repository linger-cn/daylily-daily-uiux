# 多产品线原型协作目录结构设计

## 1. 背景与目标
为了支持多个产品线的并行开发和协作，本项目需要一个清晰的目录结构来管理不同内部项目代号下的页面原型。每个产品线（项目）可能拥有独立的样式、资产和页面。

## 2. 核心架构：方案 A（资产自治型）
采用“项目代号”作为一级目录，内部资源完全自治的结构。

### 目录结构预览
```text
prototypes/
├── <project_code_1>/       # 内部项目代号 (例如: p1-daily)
│   ├── styles/             # 该项目专属的 CSS 文件
│   ├── assets/             # 该项目专属的图片、图标、插画
│   ├── index.html          # 该项目的页面目录/导航入口
│   └── <page_name>.html    # 具体原型页面 (例如: login.html)
└── <project_code_2>/       # 其他项目代号 (例如: p2-store)
    ├── styles/
    ├── assets/
    └── <page_name>.html
```

## 3. 设计规范
*   **命名约定**：
    *   项目文件夹使用内部项目代号（建议小写字母和连字符，如 `xuan-cao-v1`）。
    *   页面文件名应具有语义性（如 `login.html`, `user-profile.html`）。
*   **资源引用规范**：
    *   **私有资源**：页面应优先引用本项目 `styles/` 和 `assets/` 下的资源。
    *   **共享资源**：若需引用根目录下的全局样式（如 `themes/`），使用相对路径 `../../themes/xxx.css`。
*   **扩展性**：若单项目页面过多（>10个），建议在项目目录下创建 `pages/` 子文件夹。

## 4. 协作流程建议
1.  **初始化**：为新产品线创建对应的 `<project_code>` 文件夹。
2.  **样式定义**：在 `styles/` 下定义该项目的特有样式，或按需覆盖全局样式。
3.  **页面开发**：在项目根目录下创建 `.html` 原型文件。
4.  **入口管理**：通过项目下的 `index.html` 管理该项目的所有页面链接，方便预览。

## 5. 验收标准
*   [ ] 成功创建 `prototypes/` 根文件夹。
*   [ ] 能够通过简单的文件操作添加新的产品线。
*   [ ] 不同项目间的样式和资源不会发生冲突。
